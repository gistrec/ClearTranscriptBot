# Диаризация спикеров — ресёрч и тест

Дата: 2026-06-09. Данные на момент исследования (модели/цены могут устареть).
Решение о внедрении **не принято** — это исследование осуществимости.

## TL;DR

Пост-обработочная диаризация (вариант C) работает на реальной русской многоголосой
записи. **Merge надо делать на уровне слов, не сегментов** — иначе перекрёстная речь
схлопывается в одного спикера. Диаризация (pyannote) — главный драйвер цены/латентности,
а не дешёвый довесок. Подключать только опционально (кнопкой), чтобы не задевать
моно-записи.

## Контекст / ограничение

- Сейчас диаризация НЕ используется (нет ни флага в input whisperx, ни обработки `speaker`).
- Диаризация — это **вход** запуска, не пост-обработка по `result_json`. Поэтому кнопка
  «после результата», как у таймкодов, в лоб не работает. Обход: отдельный проход
  диаризации по сохранённому в S3 аудио + merge с текстом (вариант C), на кнопке-ре-ране.
- Полезна для длинных многоголосых (встречи/интервью, ~13% файлов ≥30 мин), вредна на
  моно (риск ложного дробления одного голоса — нулевая терпимость к false-positives).

## Модели диаризации на Replicate

| Модель | Скорость (4-мин демо) | Выход | Контроль спикеров | HF-токен |
|---|---|---|---|---|
| `meronym/speaker-diarization` | ~14 с | URL на output.json | нет (автомат) | **не нужен** |
| `collectiveai-team/speaker-diarization-3` | ~131 с (×9) | inline JSON | `num/min/max_speakers` | — |
| whisperx native (`diarization:true`) | в одном проходе | слова с метками | `min/max_speakers` | **нужен** |

Нюанс формата: meronym и collectiveai отдают `start/stop` строками `H:MM:SS.ffffff`
(не float) — парсить через `_to_seconds()`.

## Тест на реальной записи

- Файл: `59034_Компания.ogg`, ~26 мин, банковские переговоры (id 17128 в БД).
- whisperx prediction `ftae3r3r6drnc0cyn858s9ddg0` (без align), `evn91jfqn1rnc0cyn8a8fshh7m` (с align).
- meronym prediction `n094pre8chrgc0cyn859h5nhq8` → нашёл **6 спикеров**, 141 интервал.

### Сегментный merge vs пословный

| | Сегментный | Пословный (`align_output:true`) |
|---|---|---|
| Спикеров сматчено | 5/6 | **6/6** |
| Блоков | 62 (слепленные) | **92** |
| Перекрёстная речь | схлопывается в доминирующего | **разносится корректно** |

Пословный merge разносит перебивки («Угу, по…», «как…») и Q&A клиент↔менеджер по
ролям — пригодный к выдаче результат. Артефакт: на границах смены спикера 1–2 слова
иногда уезжают к мгновенно перекрывшему голосу (косметика; лечится пост-фильтром
«блоки короче N слов вливать в соседний доминирующий спикер»).

## Стоимость / скорость (26-мин файл)

| Шаг | predict_time | ₽ (прикидка) |
|---|---|---|
| whisperx без align | 19.4 с | ~2.17 ₽ (A100) |
| whisperx с `align_output` | 33.1 с | ~3.7 ₽ (+70%) |
| meronym диаризация | 88.8 с | ~1.6 ₽ (T4) … ~9.9 ₽ (A100) — GPU API не отдаёт |

Вывод: pyannote **не батчуется** как whisperx → диаризация в ~4.5× дольше транскрипции и
доминирует в стоимости. В абсолюте единицы–десяток ₽, и только по кнопке — приемлемо.

## Алгоритм merge (валидированный)

1. whisperx с `align_output:true` → сегменты со словами `{word, start, end}`.
2. Диаризация → интервалы `{speaker, start, end}` (нормализовать clock-строки → секунды).
3. Каждому **слову** присвоить спикера по максимальному перекрытию интервалов;
   слова без таймстампа/перекрытия — нести спикера предыдущего слова (carry-forward).
4. Сгруппировать подряд идущие слова одного спикера в блоки → `[Спикер N] [mm:ss] текст`.

## Пути реализации (оба on-demand, без вопроса до старта)

1. **C: meronym + наш пословный merge** — HF-токен не нужен; кнопка ре-ранит whisperx с
   `align_output` + meronym, склеиваем. Минус: держим merge-код + парсер clock-строк.
2. **whisperx-native** — один ре-ран с `diarization:true`+`align_output:true`, слова уже с
   метками спикера, merge-код не нужен. Минус: нужен HF-токен в боте.

Рекомендация: **не** включать `align_output` на всех задачах (налог на ~7500 прогонов,
бóльшая часть которых диаризацию не запросит) — только на кнопочном ре-ране.
